# Implementierungsplan: ios-045

Ticket: [ios-045](../ios/ios-045-idle-ring-thin.md)
Erstellt: 2026-05-15

---

## Ziel in einem Satz

Der Idle-Ring (`BreathDial`) wird visuell auf die Running-Sprache angeglichen — duenner Track + duenner Bogen + kleiner Bead in identischen Werten wie `BreathingCircleView`, ohne die Drag-/Geometrie-/Accessibility-Logik anzufassen.

---

## Annahmen

Bewusste Festlegungen, die in den Plan eingeflossen sind. Bitte beim Review pruefen.

- **Bead-Ruhegroesse: 9 pt Durchmesser** — exakt wie `BreathingCircleView.dotSize`.
- **Bead-Drag-Groesse: 14 pt Durchmesser** (~55 % groesser als Ruhegroesse). Begruendung: Handoff schlaegt r=5→r=7 vor (40 %), unsere Baseline ist 9 statt 10, deshalb leicht hoeher angesetzt, damit der Drag-Feedback spuerbar bleibt. Animation: `.easeOut(duration: 0.15)` analog zum bestehenden Drag-Feedback.
- **Hit-Area-Padding: 24 pt radial nach aussen** — direkt aus dem Handoff (`Circle().inset(by: -24)`).
- **Bead-Farbe und Shadow: identisch zu Running** — `theme.interactive` Fill, `.shadow(color: theme.interactive.opacity(0.6), radius: 4)`.
- **Reduced Motion: kein Sonderfall noetig**. Es bleibt keine kontinuierliche Animation uebrig (Halo entfaellt vollstaendig). Der Bead-Grow ist eine Direct-Touch-Reaktion (vergleichbar mit Button-Press) und wird auch bei Reduced Motion belassen — Apple HIG behandelt solche Mikro-Feedbacks nicht als Reduced-Motion-relevant.
- **`theme.dialActiveArc` vs. `theme.interactive`**: BreathDial behaelt `theme.dialActiveArc` (der Token loest aktuell auf `theme.interactive` auf — identisch). Ein eigener semantischer Token bleibt sinnvoll, falls eine zukuenftige Palette Idle vs. Running unterschiedlich tonen will. Wir aendern nichts an den Theme-Tokens, ausser dem Cleanup unten.
- **Theme-Token-Cleanup: `theme.dialDropletCore` und `theme.dialDropletHalo` werden entfernt**, sobald `BreathDial` sie nicht mehr nutzt. Beide werden ausschliesslich in `BreathDial.swift` und in `ThemeColors.swift` referenziert (Stand: aktuelle Recherche). Risiko gering.
- **Diameter aus `TimerView` bleibt unveraendert**: Der Idle-Ring nutzt weiterhin den vom Parent uebergebenen `diameter` (180–220 pt). Nur die Strichstaerke aendert sich. Der Bead wird auf der gleichen Bahn (`ringRadius = (diameter - ringWidth) / 2`) positioniert — Bahn verschiebt sich minimal nach aussen, weil ringWidth von ~16 pt auf 3 pt schrumpft.

## Design-Entscheidungen

### 1. Zentrale Konstanten in `RingMetrics`

**Trade-off:** Eine separate Datei macht den Plan etwas groesser, garantiert aber, dass Idle ↔ Running garantiert synchron bleiben. Der Ticket-Hinweis sagt explizit: „Wenn der Code spaeter die Werte zentral aendert, ziehen Idle und Running zusammen mit."

**Entscheidung:** Eigene Datei `RingMetrics.swift` mit drei `static let`-Konstanten:

```swift
enum RingMetrics {
    static let lineWidth: CGFloat = 3
    static let beadDiameter: CGFloat = 9
    static let beadShadowRadius: CGFloat = 4
}
```

`BreathDial` und `BreathingCircleView` lesen daraus. Damit ist die visuelle Einheit auch in Zukunft erzwungen.

### 2. Bead-Grow auf Drag — eigenes `@State`

**Trade-off:** Wir koennten den Bead-Grow ueber den Wert (z.B. „bead wird groesser, wenn sich `value` schnell aendert") triggern oder ueber einen expliziten Drag-State. Der Wert-basierte Ansatz ist unzuverlaessig (Drag bleibt auch still moeglich).

**Entscheidung:** `@State private var isDragging: Bool` — wird in `DragGesture.onChanged` auf `true`, in `DragGesture.onEnded` auf `false` gesetzt. Bead-Frame: `isDragging ? RingMetrics.beadDiameter * 1.55 : RingMetrics.beadDiameter`, animiert mit `.easeOut(duration: 0.15)`.

### 3. Hit-Area ausserhalb der View-Frame

**Trade-off:** `contentShape(Circle().inset(by: -24))` extendet die Hit-Area ueber den Frame hinaus. Alternativ koennte man den ganzen Dial in einen 24-pt-groesseren Frame packen und alles intern zentrieren — das wuerde aber das Layout in `TimerView` veraendern (mehr Platz noetig).

**Entscheidung:** `contentShape` mit negativem Inset. Layout in `TimerView` bleibt unveraendert. Die Hit-Erweiterung ist rein logisch, keine visuelle Aenderung.

## Refactorings

1. **`RingMetrics.swift` als neue geteilte Konstanten-Datei.** Keine Logik, nur drei Konstanten. Risiko: minimal. Beide Komponenten lesen daraus.
2. **`BreathingCircleView.lineWidth` und `.dotSize` durch `RingMetrics` ersetzen.** Risiko: minimal. Existierende Tests (UI-Tests, Screenshots) verlassen sich nicht auf konkrete Pixelwerte.
3. **Theme-Token-Cleanup (`dialDropletCore`, `dialDropletHalo`).** Risiko: minimal — beide Tokens werden ausschliesslich in `BreathDial.swift` benutzt; nach Migration kann der Token-Block weg.

Kein anderes Refactoring noetig. `BreathDialGeometry` bleibt unveraendert (Drag-Mathematik ist orthogonal zur Ring-Optik).

## Fachliche Szenarien

### AK-1: Idle-Ring sieht aus wie der laufende Ring

- Gegeben: Timer-Tab im Idle-Modus geoeffnet
  Wenn: User sieht den Dauer-Picker
  Dann: Track-Stroke ist 3 pt, Bogen-Stroke ist 3 pt mit rundem Ende, Track-Farbe ist die gleiche, Akzent-Farbe des Bogens ist die gleiche wie im laufenden Timer

- Gegeben: Timer mit „Beginnen" gestartet
  Wenn: Visueller Uebergang vom Idle-Ring zum laufenden Ring (BreathingCircleView)
  Dann: Beide Ringe wirken aus derselben Familie — gleiche Strichstaerke, gleiche Farben

### AK-2: Bead ist klein und in Akzent-Farbe

- Gegeben: Idle-Ring im Ruhezustand
  Wenn: User schaut auf die Bead-Position (entspricht aktuellem Minuten-Wert)
  Dann: Bead ist ein 9-pt-Kreis in Akzent-Farbe (`theme.interactive`), mit weichem Shadow (Radius 4) — kein umrahmter Tropfen, kein Halo

### AK-3: Bead vergroessert sich beim aktiven Drag

- Gegeben: Bead in Ruhegroesse (9 pt)
  Wenn: User legt Finger auf den Ring und beginnt zu ziehen
  Dann: Bead waechst sichtbar auf ~14 pt, Wachstum mit easeOut-Animation

- Gegeben: Bead in vergroessertem Zustand
  Wenn: User hebt den Finger
  Dann: Bead schrumpft zurueck auf 9 pt

### AK-4: Touch-Area reicht ueber den Ring hinaus

- Gegeben: Idle-Ring mit duennem 3-pt-Stroke und kleinem 9-pt-Bead
  Wenn: User tippt 20 pt ausserhalb des sichtbaren Rings (oberhalb der Glyphenmitte)
  Dann: Drag-Geste startet trotzdem, Wert aendert sich

- Gegeben: User tippt im Zentrum der Dial (in der Naehe der Zahl)
  Wenn: User beginnt Drag-Geste
  Dann: Geste wird ignoriert (bestehender Mittel-Gate via `sqrt > ringRadius * 0.5` bleibt)

### AK-5: Idle-Ring atmet nicht

- Gegeben: Timer-Tab im Idle, User wartet ohne Interaktion
  Wenn: User schaut auf den Ring ueber 8 s
  Dann: Keine sichtbare Bewegung — keine Pulsation, kein Glow, keine Skalierung

### AK-6: Drag/Klemmung/Mittel-Anzeige unveraendert

- Gegeben: BreathDial mit value=18
  Wenn: User dragt zur 3-Uhr-Position
  Dann: Wert springt auf 15, Mittel-Text zeigt „15 Minuten"

- Gegeben: BreathDial mit value=1
  Wenn: User dragt auf die 12-Uhr-Position (Raw-Winkel = 0)
  Dann: Wert bleibt 1 (Klemmung greift)

### AK-7: VoiceOver-Slider unveraendert

- Gegeben: VoiceOver aktiv, Fokus auf dem Dial
  Wenn: User wischt nach oben
  Dann: Wert erhoeht sich um 1, VoiceOver liest „X Minuten" vor

- Gegeben: Wert bei 60
  Wenn: User wischt weiter nach oben
  Dann: Wert bleibt 60 (Klemmung)

### AK-8: Reduced Motion ohne Auswirkung

- Gegeben: Reduced Motion in iOS-Einstellungen aktiv
  Wenn: User oeffnet Timer-Tab im Idle
  Dann: Idle-Ring sieht identisch aus wie mit Reduced Motion aus — keine sichtbare Aenderung (weil ohnehin keine Atem-Animation existiert)

## Risiken

| Risiko | Mitigation |
|---|---|
| Bestehende UI-Tests (`timer.dial`, `timer.dial.value`) brechen | Accessibility-Identifier strikt unveraendert lassen. Bei Lauf des UI-Tests einmal manuell verifizieren. |
| `contentShape(Circle().inset(by: -24))` ausserhalb des Frames triggert kein Drag (SwiftUI clippt evtl.) | Falls beobachtet: Fallback auf groesseren Frame fuer den Dial-Wrapper oder `.contentShape` ueber `.padding(-24)` Trick. Manueller Test (Ticket-Schritt 4) deckt das ab. |
| Theme-Token-Cleanup uebersieht Verwendung | Pre-Commit-Check: `grep -r "dialDropletCore\|dialDropletHalo" ios/` — darf nach Cleanup keine Treffer mehr liefern. |
| Bead waehrend Wert-Change-Animation springt | Bestehende `.animation(.easeOut(duration: 0.15), value: self.value)` auf Bead-Offset belassen. Neue Animation fuer `isDragging` ist auf einem anderen Modifier — keine Interferenz. |
