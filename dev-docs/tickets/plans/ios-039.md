## Analyse

### 3 Schreibstellen in TimerViewModel → settingsRepository

1. **Zeile 186** — `updateFromPraxis()`: Schreibt Settings nach Praxis-Wechsel ins alte Repository. Redundant, da Praxis bereits die Quelle ist.
2. **Zeile 507** — `executeSaveSettings()`: Wird vom Reducer-Effect `.saveSettings` aufgerufen beim Timer-Start. Schreibt Settings ins alte Repository.
3. **Zeile 516** — `saveSettings()`: Oeffentliche Methode. Wird von **keiner View** aufgerufen (Grep bestaetigt). Toter Code.

### Reducer-Effect `.saveSettings`

`TimerReducer.reduceStartPressed()` (Zeile 79) erzeugt `.saveSettings(updatedSettings)` — dieser Effect aktualisiert `durationMinutes` in den Settings. Aber: `executeStartTimer()` (Zeile 480-484) speichert `durationMinutes` bereits via `praxisRepository.save()`. Der Settings-Effect ist also doppelt.

### UserDefaultsTimerSettingsRepository — Entfernen oder behalten?

**Behalten**, aber nur als interne Dependency von `UserDefaultsPraxisRepository`. Der Migrations-Pfad in `UserDefaultsPraxisRepository.load()` (Zeile 46-51) liest via `settingsRepository.load()` aus den alten UserDefaults-Keys. Dieser Pfad ist fuer bestehende User noetig, die noch keine Praxis-Daten haben.

Das `TimerSettingsRepository`-**Protocol** im Domain-Layer kann aber entfernt werden — es wird nur noch als konkreter Typ (`UserDefaultsTimerSettingsRepository`) innerhalb von `UserDefaultsPraxisRepository` benoetigt. Kein externer Code braucht die Abstraktion mehr.

### hasSeenSettingsHint-Tests

3 Tests in `TimerViewModelSettingsTests.swift` (Zeilen 510-546) testen direkt `UserDefaults.standard.bool(forKey: "hasSeenSettingsHint")`. Es gibt keine UI die diesen Key liest oder setzt — das Feature wurde nie gebaut. Tests entfernen.

## Fachliche Szenarien

### AK-1: TimerViewModel schreibt nicht mehr in UserDefaultsTimerSettingsRepository

- Gegeben: TimerViewModel mit PraxisRepository
  Wenn: Timer gestartet wird
  Dann: Nur PraxisRepository wird geschrieben, nicht TimerSettingsRepository

- Gegeben: Praxis wird gewechselt (updateFromPraxis)
  Wenn: Neue Praxis wird geladen
  Dann: Settings werden nur in-memory aktualisiert, kein Schreibzugriff auf TimerSettingsRepository

### AK-2: TimerSettingsRepository-Dependency aus TimerViewModel entfernt

- Gegeben: TimerViewModel init-Signatur
  Wenn: Instanz erstellt wird
  Dann: Kein `settingsRepository`-Parameter mehr vorhanden

### AK-3: hasSeenSettingsHint-Tests entfernt

- Keine fachlichen Szenarien — reine Test-Entfernung.

### AK-4: UserDefaultsTimerSettingsRepository bleibt fuer Migration

- Gegeben: Bestehender User ohne Praxis-Daten, aber mit alten MeditationSettings-Keys
  Wenn: App startet und `UserDefaultsPraxisRepository.load()` aufgerufen wird
  Dann: Migration liest aus alten Keys und erstellt eine Praxis
