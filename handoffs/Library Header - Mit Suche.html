Still Moment · Library-Header · Mit Suche

Drei Pillen. Drei Screens.

Suchen, Hinzufügen, Info — jede Aktion ist eine eigene Pille im Header, jede Pille öffnet einen eigenen Screen. Klare Navigations-Map ohne Inline-Modi, jede Route mit ihrem eigenen State, ihrem eigenen Layout.

Entscheidung

Drei Pillen-Buttons — jede ist eine Navigation.

Im Library-Header sitzen drei eigenständige Pillen: »Suchen« links, »+« und »i« rechts. Tap auf eine führt zu einem eigenen Screen — kein Inline-Mode, kein State-Wechsel im selben View. Die Suchpille ist im Idle ein Button, der wie ein Suchfeld aussieht: Lupe, »Suchen«-Label, Capsule. Erst auf dem Such-Screen ist sie ein echtes TextField mit Fokus, Caret, Filter.

Einstellungen
01

Suchpille ist im Idle ein Button, kein Eingabefeld.

Sie sieht aus wie ein Suchfeld (Lupe, „Suchen“-Label, Capsule), aber Tap → Push auf den Such-Screen. Kein Fokus, kein Caret, kein Filter auf dieser Ebene. Macht das Verhalten deterministisch über alle iOS-Versionen — und entkoppelt Library-Layout von Search-Layout.

02

Search-Screen ist eigene Route, kein Inline-Mode.

Tap auf die Suchpille pusht einen SearchView. Der hat sein eigenes Layout, seinen eigenen State, sein eigenes Filter-Modell. Rechts oben „Abbrechen“ popt zurück. Vorteil: Library und Suche können sich unabhängig weiterentwickeln — Filter, Tags, Voice-Search nur in SearchView, nicht in der Library.

03

Flache Trefferliste, Lehrer im Subtitle.

Im Such-Modus wird nicht nach Lehrer gruppiert — das wäre redundant zur Standard-Bibliothek. Stattdessen flache Liste mit Titel (Newsreader 15) und Lehrer · Dauer als Untertitel. Match-Highlight greift in beiden Feldern. Oben Eyebrow 4 Treffer in tracked caps.

SwiftUI-Skizze · für Claude Code Pattern

Kein .searchable(). Stattdessen TextField direkt in der Capsule, gebunden an einen @State var query. Der Filter läuft in einer filteredTracks-Computed-Property — kein Backend, kein Submit-Roundtrip. 

struct LibraryHeader: View {
    @Binding var query: String
    let onAdd: () -> Void
    let onInfo: () -> Void
    @FocusState private var searchFocused: Bool

    var body: some View {
        HStack(spacing: 10) {
            // Search-Pille — flex, füllt verfügbaren Raum
            HStack(spacing: 10) {
                Image(systemName: "magnifyingglass")
                    .foregroundStyle(searchFocused ? Color.accent : .secondary)

                TextField("Suchen", text: $query)
                    .textStyle(.body)
                    .focused($searchFocused)
                    .submitLabel(.search)

                if !query.isEmpty {
                    Button { query = "" } label: {
                        Image(systemName: "xmark.circle.fill")
                            .foregroundStyle(.tertiary)
                    }
                    .transition(.scale.combined(with: .opacity))
                }
            }
            .padding(.horizontal, 14)
            .frame(height: 40)
            .background(.regularMaterial, in: Capsule())
            .overlay {
                if searchFocused {
                    Capsule().stroke(.accent.opacity(0.15), lineWidth: 1.5)
                }
            }
            .animation(.snappy(duration: 0.2), value: query.isEmpty)

            // Action-Pille — bleibt wie heute
            LibraryActionPill(onAdd: onAdd, onInfo: onInfo)
        }
        .padding(.horizontal, 22)
        .padding(.top, 12)
        .padding(.bottom, 8)
    }
}

Einbau im Library-Screen: das ist genau das View, das via .safeAreaInset(edge: .top) über dem ScrollView hängt — wie im vorigen Vorschlag. query wandert per Binding in das Parent-ViewModel, und filteredTracks ist ein Computed über tracks.filter { … }.
